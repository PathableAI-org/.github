# Support for PathAble Projects

This document describes the default support expectations for repositories
maintained by PathAble.

Each repository may define its own support policy based on its purpose,
maturity, audience, and maintenance capacity. Repository-specific guidance
takes precedence over this organization-wide policy.

## Repository Status

Before using or requesting support for a project, review its `README.md` for:

- Purpose and intended audience
- Current maturity or stability
- License and permitted use
- Installation and usage instructions
- Known limitations
- Repository-specific support channels

Experimental, archived, reference, or pre-release projects may receive limited
maintenance and may change without notice.

## Appropriate GitHub Issues

When enabled by a repository, GitHub issues may be used for:

- Reproducible software defects
- Documentation errors or unclear instructions
- Accessibility barriers in the repository or software
- Focused feature proposals
- Questions directly related to using or contributing to the repository

Before opening an issue:

1. Read the repository documentation.
2. Search existing issues for related reports.
3. Confirm that the repository accepts the proposed issue type.
4. Remove all private, sensitive, or identifying information.
5. Provide a minimal reproduction using synthetic data when possible.

## Accessibility Feedback

Accessibility feedback is welcome and should be treated as substantive
technical feedback.

When reporting an accessibility barrier, include:

- The task you were attempting
- What prevented or complicated completion
- The relevant page, component, command, or documentation
- Browser, operating system, assistive technology, or input method when useful
- A suggested improvement, if you have one

You are not required to disclose a disability, diagnosis, or personal medical
information to report an accessibility problem.

## GitHub Is Not a Customer-Support Channel

Public GitHub issues must not be used for:

- Customer account or billing questions
- Production incidents or service outages
- Participant-specific or agency-specific problems
- Contractual support requests
- Time-sensitive operational assistance
- Personal accommodation requests containing private information
- Security vulnerabilities
- Code of Conduct reports

Customers and partners should use the support channels established through
their PathAble agreement, onboarding materials, or designated contacts.

## Sensitive Information

Never include the following in a public issue, discussion, pull request,
screenshot, attachment, or log:

- Protected health information
- Participant, customer, partner, or employee information
- Credentials, tokens, secrets, or private keys
- Production data
- Private endpoints or confidential architecture
- Security vulnerabilities or exploit details
- Private communications or contractual information

Use clearly synthetic names and data in all examples.

If sensitive information is posted accidentally, do not repeat it elsewhere.
Notify a repository maintainer privately through an established channel.

## Security Reports

Do not report security vulnerabilities through public GitHub issues.

Follow the affected repository's `SECURITY.md` or private reporting
instructions. If no security-reporting process is published, do not disclose
the vulnerability publicly while attempting to identify the appropriate
private contact.

## Response Expectations

PathAble reviews public repository activity as capacity permits.

Unless a repository or customer agreement explicitly states otherwise:

- No response time is guaranteed.
- Opening an issue does not guarantee investigation or implementation.
- Feature requests may be declined or deferred without a delivery date.
- Maintainers may close issues that are out of scope, incomplete, inactive,
  unsafe to discuss publicly, or inconsistent with project direction.
- Community support is provided on a best-effort basis.

Paid or contractual support commitments are governed by the applicable
agreement, not this file.

## Contributions

A proposed fix may be welcome when the repository accepts contributions.
Review the organization and repository-specific `CONTRIBUTING.md` files before
starting work.

Please discuss substantial changes with maintainers before investing
significant effort.

## Conduct

Participation in PathAble repositories is governed by the applicable
`CODE_OF_CONDUCT.md` and repository-specific community rules.
